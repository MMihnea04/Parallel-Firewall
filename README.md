# Parallel Firewall

This project implements a parallel firewall simulator using POSIX threads. It is designed as an educational exercise focused on parallel programming, thread synchronization, and transforming a serial application into a correct and efficient multithreaded one. The firewall processes synthetic network packets generated by a producer thread and analyzed by multiple consumer threads, with results written to a log file in deterministic order.

The emphasis of the project is correctness, synchronization, and respecting concurrency constraints rather than raw performance.

---

## Overview

A firewall inspects network packets and decides whether they should be allowed to pass or be dropped based on a set of rules. In a real system, packets originate from a network interface and contain protocol headers and payloads. In this assignment, packets are synthetic data structures composed of a source identifier, a destination identifier, a timestamp, and a payload.

Packets are generated by a producer thread and inserted into a shared ring buffer. Multiple consumer threads retrieve packets from this buffer, apply firewall rules, and record the result of the decision. The final output is a log file containing the firewall decision for each packet, ordered by packet timestamp.

---

## Architecture

The application is following a producer to consumer design.

A single producer thread reads packets from an input file and enqueues them into a shared circular buffer. Multiple consumer threads dequeue packets from this buffer, process them independently, and store the results. Synchronization is handled using mutexes and condition variables from the POSIX threads API, ensuring that no busy waiting occurs.

The program guarantees that consumers block when no packets are available and that the producer blocks when the buffer is full.

---

## Ring Buffer Design

The core shared data structure is a ring buffer implemented as a fixed-size circular array. It maintains separate read and write positions, along with a current length and total capacity. The buffer supports concurrent access by one producer and multiple consumers.

Access to the ring buffer is protected by a mutex. Two condition variables are used to coordinate access: one to signal when the producer can enqueue new data and one to signal when consumers can dequeue data. The buffer also supports a stop mechanism that allows consumers to exit cleanly once the producer has finished publishing packets.

Each packet dequeued from the buffer is assigned a monotonically increasing sequence number, which is later used to ensure deterministic ordering of results.

---

## Consumer Threads

Each consumer thread repeatedly dequeues packets from the ring buffer. For every packet, the consumer applies the firewall logic, computes a hash over the packet contents, and records the action taken together with the packet timestamp.

Consumers do not write directly to the output file. Instead, they store results in a shared results array indexed by packet sequence number. This design avoids race conditions on file I/O and allows results to be written in a deterministic order.

When the producer signals that no more packets will be produced, consumers exit gracefully.

---

## Packet Processing

The firewall logic checks packet source addresses against a predefined set of allowed ranges. If the source address falls within one of these ranges, the packet is accepted; otherwise, it is dropped.

A packet hash is computed using a deterministic hashing function that iterates multiple times over the packet data. This hash, together with the firewall decision and timestamp, is used for logging and validation.

---

## Log File Generation

The final output of the firewall is a log file containing one line per packet. Each line includes the firewall decision, the packet hash, and the packet timestamp.

Although packet processing is parallel and non-deterministic, the log file must be written in ascending order of packet timestamps, matching the order in which packets were produced. This ordering is enforced during processing by indexing results using sequence numbers, not by sorting the log after processing completes.

Once all consumer threads have finished, the main thread writes the collected results to the output file in order.

---

## Building and Running

Both the serial and parallel versions of the firewall can be built using `make` from the `src` directory. This produces two binaries: `serial` and `firewall`.

The parallel firewall is executed by providing an input file, an output file, and the number of consumer threads. At least one producer thread and the specified number of consumer threads will be created.

When a single consumer is specified, the program falls back to the serial implementation.

---

## Testing and Validation

Automated tests are provided in the `tests` directory. These tests compare the output of the parallel implementation against the serial version for correctness. The test suite checks various packet counts, thread counts, and ordering requirements.

A grading script runs all tests and assigns points based on correctness and coding style. Linters are also used to enforce coding standards.

---

## Synchronization Constraints

Busy waiting is strictly forbidden. Threads must block using proper synchronization primitives when waiting for buffer space or data availability.

The number of running threads must be at least the number of consumer threads plus one producer thread. All shared data structures must be accessed in a thread-safe manner.

---

## Limitations

This firewall is a simulation and does not process real network packets. It does not support dynamic reconfiguration of rules or runtime packet injection. The design is intended for educational use and experimentation with multithreading concepts.
